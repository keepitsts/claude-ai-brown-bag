# Stop Coding

## Slide

**Title:** Stop Coding

- Your hands are no longer the best tool for writing code. **Models running on supercomputers are faster, more consistent, and don't get tired**
- Engineering was never about typing — it was about solving problems. Now you solve problems by **directing agents instead of typing syntax**
- Your new workflow: **prompt, review, refine** — not open file, type code, debug, repeat
- This doesn't mean you stop thinking. It means you **stop being the bottleneck** between your ideas and working software
- Go all in. Half-measures don't work — if you're still hand-writing boilerplate, you're leaving 10x on the table

---

## Notes

### The Mindset Shift

This is the hardest tactic because it's not a technique — it's a habit change. Engineers have spent years building muscle memory for writing code. The instinct when you see a bug is to open the file and start typing. That instinct is now a bottleneck.

**The old workflow:**
1. Read the code
2. Understand the problem
3. Think about the fix
4. Type the fix
5. Run the build
6. See the error
7. Go back to step 3

**The new workflow:**
1. Tell the agent what's wrong
2. The agent reads the code, understands the problem, thinks about the fix, types the fix, runs the build, sees the error, and fixes it again
3. You review the result

Steps 1-6 of the old workflow are now steps the agent does in its loop. You do steps 1 and 3 of the new workflow — direction and review.

### What "Stop Coding" Actually Means

It does NOT mean:
- Stop understanding code
- Stop reading code
- Stop making architectural decisions
- Stop reviewing what the agent produces
- Stop thinking about engineering problems

It DOES mean:
- Resist the urge to type code manually when an agent can do it
- Stop writing boilerplate by hand (CRUD routes, type definitions, test scaffolding)
- Stop context-switching between "thinking about the problem" and "typing the solution"
- Let the agent handle the mechanical translation from intent to implementation

### Daily Actions

- Resist the urge to type code manually — describe what you want instead
- Go all in on agentic coding tools — Claude Code, claude.ai, the full pipeline
- Don't code, write prompts and review output
- Communicate to your agents what you want built — be an architect, not a typist

### Why Half-Measures Don't Work

Engineers who use AI tools "sometimes" get marginal benefit. The compounding effect comes from:
- Building up CLAUDE.md with real conventions (only happens if you use it daily)
- Learning what makes a good prompt (only happens through repetition)
- Training your instinct to direct instead of type (only happens if you commit)
- Building a library of reusable prompts and templates (only happens over time)

The engineer who goes all in for a month learns 10x more than the one who tries it occasionally for six months.
